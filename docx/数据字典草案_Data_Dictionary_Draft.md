#### Sheet 1-1: `合同支付情况`（28 行 × 41 列）

**表头结构（3 层合并单元格）**:

```
Row 1:  A1:AO1 合并 → "合同支付情况"（全表标题）

Row 2:  A2:A3 合并 → "序号"
        B2:B3 合并 → "编号"
        C2:C3 合并 → "供应商"
        D2:D3 合并 → "合同内容"
        E2:E3 合并 → "签订日期"
        F2:F3 合并 → "合同金额"
        G2:K2 合并 → "支付节点"        ← 父级分类头
        L2:O2 合并 → "累计支付"        ← 父级分类头
        P2:AO2 合并 → "支付明细"       ← 父级分类头（24 个月份列）

Row 3:  G="施工前阶段"    H="施工中阶段"    I3:K3 合并 → "竣工阶段"
        J, K 为竣工阶段的附加子列（部分合同有 4-5 个付款阶段）
        L="累计已付\n万元"  M="剩余未付\n万元"  N="已付比例"  O="剩余未付比例"
        P~AO = 24 个 Excel 日期序列值，代表逐月的支付明细列：
          45962, 45992, 46023, 46054, 46082, 46113,
          46143, 46203, 46204, 46235, 46266, 46296,
          46327, 46357, 46388, 46419, 46447, 46478,
          46508, 46539, 46569, 46600, 46631, 46661,
          46692, 46722
```

**列级字段定义（Col A ~ Col AO）**:

| 列 | Excel 表头文字 | 建议字段名 | 类型 | 说明 |
|----|---------------|-----------|------|------|
| A | 序号 | `seq` | INT | 1~25 |
| B | 编号 | `contract_no` | VARCHAR(20) | **合同唯一编号**，格式 CX001~CX018 |
| C | 供应商 | `supplier_name` | VARCHAR(200) | 承包/供应单位全称 |
| D | 合同内容 | `contract_desc` | VARCHAR(500) | 合同类型/内容描述 |
| E | 签订日期 | `sign_date` | DATE | 如 2024-08-19 |
| F | 合同金额 | `contract_amount` | DECIMAL(18,4) | ⚠ 当前数据行中此列全为空，待业务人员补填 |
| G | 施工前阶段 | `pre_constr_terms` | TEXT | 施工前/预付款阶段支付条款 |
| H | 施工中阶段 | `mid_constr_terms` | TEXT | 施工中期支付条款 |
| I | 竣工阶段 | `completion_terms` | TEXT | 竣工阶段支付条款 |
| J | （竣工阶段子列） | `completion_sub_terms` | TEXT | 竣工后的附加阶段（如结算评审后支付至90%） |
| K | （尾款/质保阶段） | `warranty_terms` | TEXT | 质保金/尾款支付条款 |
| L | 累计已付（万元） | `cumulative_paid` | DECIMAL(18,4) | |
| M | 剩余未付（万元） | `remaining_unpaid` | DECIMAL(18,4) | |
| N | 已付比例 | `paid_ratio` | VARCHAR(20) | ⚠ 含 `#DIV/0!` 公式错误值 |
| O | 剩余未付比例 | `remaining_ratio` | VARCHAR(20) | ⚠ 含 `#DIV/0!` 公式错误值 |
| P~AO | 日期序列值(标题) / 当月支付金额(数据) | `payment_amount` | DECIMAL(18,4) | 24 个月份列，标题为 Excel 日期序列值 |

> 📅 **日期序列值转换**: 45962 → 2025-11-01, 45992 → 2025-12-01, ... 46722 → 2027-11-01。公式: `DATE(1899,12,30) + serial_number`

**样本数据（Row 4, CX001）**:
```
序号=1, 编号=CX001, 供应商=四川锦鑫川荣工程咨询有限公司,
合同内容=四川省国家工程投资建设委托招标代理, 签订日期=2024-08-19,
施工前阶段=招标代理服务费在乙方完成合同约定代理事项后10个工作日内一次性支付...
施工中阶段=*, 竣工阶段=*, 累计已付=0, 剩余未付=0, 已付比例=#DIV/0!
```

**样本数据（Row 8, CX005，含多阶段付款）**:
```
序号=5, 编号=CX005, 供应商=中储粮成都储藏研究院有限公司,
合同内容=建设工程勘察、初设设计服务合同, 签订日期=2025-08-11,
施工前阶段=合同签订后30日内，支付合同暂定设计价的20%(预付款)
施工中阶段=提交方案设计资料并通过审批后，支付暂定设计价的30%
竣工阶段=提交初设资料通过评审且取得概算审核批复后，支付至80%
```

## 三、跨文件字段关联分析

### 3.1 确认为关联键的字段

| 字段名（Excel 原文） | 出现位置 | 建议关联方式 |
|---------------------|---------|-------------|
| **项目名称** | 文件 5（新投/续投）、文件 6、文件 7 | **核心 JOIN 键**。建议创建 `project_master` 表，以项目名称为自然键或生成 `project_id` 代理键 |
| **业主单位** | 文件 5（新投/续投）、文件 6、文件 7 | FK → `dict_owner_unit` |
| **编号**（合同编号） | 文件 1（合同支付情况 + 产值情况） | 合同表 `contract` 的业务主键。两个 Sheet 通过此字段 1:1 关联 |
| **供应商** | 文件 1 的两个 Sheet | FK → `dict_supplier` |
| **签订日期** | 文件 1 的两个 Sheet | 合同日期，两个 Sheet 值一致 |
| **形象部位** | 文件 4（周报） | FK → `dict_location` |

### 3.2 隐式关联

以下文件虽无显式的"项目名称"列，但**整个文件服务于同一项目**：

- 文件 1（合同台账）→ 川西项目
- 文件 2（进度款审核）→ 川西项目
- 文件 3（周报百分比）→ 川西项目
- 文件 4（周报描述）→ 川西项目

> **设计建议**: 所有表均加入 `project_id` 外键，显式关联。

### 3.3 非关联字段（同名但含义不同）

以下字段名在不同文件中出现但**业务含义完全不同**，不应建立关联：

- **序号** — 出现在 5 个文件中，但纯属各自的行编号，不构成跨表 JOIN
- **备注** — 不同文件中的备注语义不同（合同备注 vs 项目级别备注）
- **日期** — 文件 1 的 Sheet1 中是"记录日期"（如"3月22"），文件 1 的合同台账中是"签订日期"

## 五、类型不一致与数据质量问题

| 问题 | 文件/位置 | 详情 | 建议处理 |
|------|----------|------|---------|
| 金额单位不一致 | 文件 1(万元) vs 文件 2(元) | 合同台账用万元，进度款审核用元 | 建表统一为**元**，导入时文件 1 × 10000 |

### 6.2 核心 ER 关系

```
project_master (T1)
  ├── 1:N → contract (T2)
  │          ├── 1:N → contract_payment_milestone (T3)
  │          ├── 1:N → contract_payment_detail (T4)
  │          └── 1:1 → contract_output_value (T5)
  ├── 1:N → progress_payment_review (T6)
  ├── 1:N → weekly_progress_report (T7)
  │              └── N:1 → dict_location (T12)
  ├── 1:N → weekly_progress_metrics (T8)
  ├── 1:N → monthly_province_city_progress (T9)
  ├── 1:N → monthly_target_plan (T10)
  └── 1:N → fund_breakdown (T11)

contract (T2)
  ├── N:1 → dict_supplier (T14)

project_master (T1)
  └── N:1 → dict_owner_unit (T13)
```

### 6.3 关键设计决策

1. **`contract_payment_milestone` 建议设计为 `(contract_no, stage_order, stage_name, payment_terms)` 结构**，因为不同合同的付款阶段数不同（3~5 个阶段），不应硬编码为固定列。

2. **`progress_payment_review` 建议使用单表 + JSONB 扩展字段**，或用表继承：基础字段（序号~备注）通用，施工特有字段（建筑安装工程费、安全文明施工费等）和监理特有字段（工程监理服务费、造价咨询服务费等）存为可选列或 JSONB。

3. **所有横向月份列（文件 1 的 P~AO、文件 6 的 O~Z）必须转为纵向**（每行=一个月份），否则无法进行时间维度查询。

4. **金额统一为元**: 文件 1 和文件 5/6 的金额需 ×10000 转换。
