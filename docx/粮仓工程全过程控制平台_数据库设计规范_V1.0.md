# 粮仓工程全过程控制平台

# 数据库设计规范（V1.0）

> **配套文档说明**：本文档是《系统开发设计规范》的配套文档，专门规定业务对象（项目、合同、周报、付款、二类费用等）、原始文件、AI 解析结果、数据版本、操作日志之间的关系，以及各种约束规则。前后端开发时应以本文档为统一约束标准，后期扩展 AI 功能和新业务模块时不推翻已有数据结构。

---

## 一、设计原则

### 1.1 核心设计理念

| # | 原则 | 说明 |
|---|---|---|
| 1 | **数据底座统一** | 所有最小颗粒度数据进 `t_data_rows`，汇总、对比、报告全部基于此表 |
| 2 | **业务表与流水表分离** | 业务表（如 `t_reports`）只存"当前态"，流水表存"全部历史" |
| 3 | **四元组标记** | 每条数据必须带 `source_type` + `is_confirmed` + `source_doc` + `raw_data` |
| 4 | **金额 BIGINT（分）** | 避免 DECIMAL 浮点精度问题，前端展示时除以 100 |
| 5 | **时间 TIMESTAMPTZ** | 一律带时区 |
| 6 | **软删除优先** | 业务数据不真删，使用 `deleted_at` 标记 |
| 7 | **审计必加** | 所有写操作入 `t_audit_logs` |
| 8 | **合同版本强校验** | DB 层部分唯一索引实现 |

### 1.2 字段类型约定

| 数据类型 | 使用场景 | 约束 |
|---|---|---|
| `UUID` | 所有表的主键、外键引用 | 默认 `gen_random_uuid()` |
| `BIGINT` | 金额类字段 | 单位 = 分（前端展示 ÷ 100） |
| `INTEGER` | 序号、计数、期次 | `period_no`、`version`、`chunk_count` 等 |
| `DATE` | 日期字段 | 无时间部分 |
| `TIMESTAMPTZ` | 时间点字段 | 必须带时区 |
| `VARCHAR(N)` | 短文本 | N 根据业务合理设定上限 |
| `TEXT` | 长文本 | 合同条款、报告内容等 |
| `JSONB` | 动态结构数据 | AI 原始数据、报告负载、模板 schema |
| `BOOLEAN` | 状态开关 | `is_confirmed`、`is_active`、`is_deleted` |
| `DECIMAL(5,2)` | 百分比/比例 | 仅用于比例字段（如 `payment_ratio`） |
| `vector(1024)` | 向量嵌入 | pgvector 扩展，BGE-M3 输出 |

## 二、命名规范

### 2.1 表命名规范

| 规则 | 示例 |
|---|---|
| 系统核心表以 `t_` 为前缀 | `t_data_rows` |
| 业务用英文小写复数形式 | `t_contracts`，`t_projects` |
| 多词用下划线分隔 | `t_secondary_cost_ledger` |
| 关联/子表用父表名 + 子表名 | `t_contract_versions`，`t_contract_clauses` |
| 日志/审计表独立命名 | `t_audit_logs`，`t_ai_tasks` |

### 2.2 字段命名规范

| 规则 | 示例 |
|---|---|
| 主键统一为 `id` | `id UUID PRIMARY KEY DEFAULT gen_random_uuid()` |
| 外键统一为 `{父表单数}_id` | `project_id`，`contract_id`，`template_id` |
| 时间字段统一后辍 | `created_at`，`updated_at`，`deleted_at`，`confirmed_at` |
| 布尔字段用 `is_` / `has_` 前缀 | `is_confirmed`，`is_active` |
| 金额字段用 `_amount` 后辍 | `total_amount`，`paid_amount`，`declare_amount` |
| 类型/状态字段用 `_type` / `_status` 后辍 | `source_type`，`report_status`，`project_status` |

### 2.3 索引命名规范

| 索引类型 | 命名格式 | 示例 |
|---|---|---|
| 普通索引 | `idx_{表名}_{字段1}_{字段2}` | `idx_data_rows_project_date` |
| 唯一索引 | `idx_{表名}_unique_{字段}` | `idx_contract_version_unique` |
| 部分唯一索引 | `idx_{表名}_{描述}` | `idx_contract_one_active` |
| 外键索引 | `idx_{表名}_fk_{外键字段}` | `idx_contracts_fk_project_id` |
| 向量索引 | `idx_{表名}_embedding` | `idx_chunks_embedding` |

## 三、表关系架构

### 3.1 ER 关系总览

```
t_projects（项目主表）
    │
    ├── 1:N → t_data_rows（数据流水表） ← 系统核心数据底座
    ├── 1:N → t_contracts（合同主表）
    │          └── 1:N → t_contract_versions（合同版本表）
    │                     └── 1:N → t_contract_clauses（合同条款表）
    ├── 1:N → t_reports（报告实例表）
    │          └── N:1 → t_report_templates（报告模板表）
    ├── 1:N → t_payment_audits（进度款审核台账表）
    ├── 1:N → t_secondary_cost_ledger（二类费用台账表）
    ├── 1:N → t_knowledge_docs（知识库文档表）
    │          └── 1:N → t_knowledge_chunks（文档分块表）
    └── 1:N → t_ai_tasks（AI 任务表）

独立表（不直接关联项目）：
    t_users（用户与权限表）
    t_audit_logs（审计日志表）
```

### 3.2 核心依赖关系

```
                          ┌──────────────────┐
                          │   t_data_rows    │ ← 全系统数据底座
                          │  (单一事实来源)  │
                          └────────┬─────────┘
                                   │ 提供数据
                ┌──────────────────┼──────────────────┐
                ▼                  ▼                  ▼
         ┌──────────┐     ┌──────────┐     ┌──────────────┐
         │ t_reports │     │ t_payment│     │ 驾驶舱统计   │
         │ (报告)   │     │ _audits  │     │ (Dashboard)  │
         └──────────┘     │ (审核)   │     │ (统计服务)   │
                          └──────────┘     └──────────────┘
```

### 3.3 数据流关系

```
上传文件（MinIO）
    ↓
t_knowledge_docs（文件记录，parse_status=pending）
    ↓
AI/Tesseract/PDF 解析（异步任务）
    ↓
t_data_rows（结构化数据，is_confirmed=false，raw_data=AI原始快照）
    ↓
用户确认/修改
    ↓
t_data_rows（is_confirmed=true，confirmed_by/confirmed_at）
    ↓
业务表更新（t_contracts / t_payment_audits / t_reports 等）
    ↓
t_audit_logs（全程记录每次变更）
```

### 4.1 系统核心：数据流水表（t_data_rows）

**角色**：整个系统的事实源（Single Source of Truth）。所有工程资料最终转换为统一结构化数据存储于此表，系统全部功能均依赖此表运行。

**四元组原则**：每条数据必须带：

| 元组 | 字段 | 说明 |
|---|---|---|
| source_type | `source_type` | 施工方上报/过控审核/AI识别/系统计算 |

| source_doc | `source_doc` | 来源文档名称 |
| raw_data | `raw_data` | AI 识别原始数据快照（JSONB） |

**表结构**：

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | UUID | PK DEFAULT gen_random_uuid() | |
| project_id | UUID | NOT NULL FK→t_projects | 所属项目 |

### 4.4 合同版本表（t_contract_versions）

**角色**：合同的历史版本管理，支持合同变更多版本并行追溯。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | UUID | PK | |
| contract_id | UUID | **NOT NULL FK**→t_contracts | |
| version_no | VARCHAR(20) | NOT NULL | v1, v2, v3... |
| valid_from | DATE | NOT NULL | 生效日期 |
| valid_to | DATE | **允许 NULL** | 失效日期（NULL = 当前生效版本） |
| file_path | VARCHAR(500) | 允许 NULL | MinIO 路径 |
| total_amount | BIGINT | NOT NULL DEFAULT 0 | 合同金额（分） |
| payment_terms | JSONB | 允许 NULL | 支付条款（结构化存储） |
| change_note | TEXT | 允许 NULL | 变更说明 |
| created_at | TIMESTAMPTZ | NOT NULL | |
| created_by | UUID | FK→t_users | 创建人 |

**核心约束**：

```sql
-- 同一合同下，版本号必须唯一
UNIQUE(contract_id, version_no)

-- 同一时间只能有一个生效版本（部分唯一索引，★★★★★）
CREATE UNIQUE INDEX idx_contract_one_active
ON t_contract_versions(contract_id) WHERE valid_to IS NULL;
```

**唯一索引的意义**：保证同一合同在任何时间点只有一个"当前版本"，进度款审核时引用 `contract_version_id` 即可锁定版本。

| updated_at | TIMESTAMPTZ | NOT NULL | |

**JSON Schema 字段类型限定**（5 种）：

| type | 说明 | 示例 |
|---|---|---|
| `text` | 文本字段 | 进度描述、下周计划 |
| `number` | 数值字段 | 完成产值、工程量 |

**schema_json 示例**：

```json
{
  "fields": [
```

### 4.8 进度款审核台账表（t_payment_audits）

**角色**：进度款审核全流程记录，从申报 → AI 审核 → 人工确认 → 支付完成。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | UUID | PK | |
| project_id | UUID | **NOT NULL FK**→t_projects | |
| contract_id | UUID | **NOT NULL FK**→t_contracts | |
| contract_version_id | UUID | **NOT NULL FK**→t_contract_versions | 申请时引用的版本（锁定） |
| period_no | INTEGER | NOT NULL | 第几期次 |
| constructor_name | VARCHAR(200) | 允许 NULL | 施工单位 |
| declare_amount | BIGINT | NOT NULL | 申报金额（分） |
| ai_suggest_amount | BIGINT | 允许 NULL | AI 建议审核金额（分） |
| final_amount | BIGINT | 允许 NULL | 终审金额（分） |
| deduct_amount | BIGINT | DEFAULT 0 | 核减金额（分） |
| deduct_reason | JSONB | DEFAULT '[]' | 核减原因列表（AI 生成 + 人工补充） |
| items_json | JSONB | DEFAULT '[]' | 工程量清单明细（逐项） |
| ai_findings | JSONB | DEFAULT '[]' | AI 审核发现异常项 |
| audit_status | VARCHAR(20) | DEFAULT 'pending' | pending（待审）/auditing（审核中）/audited（已审）/paid（已付） |

| audited_by | UUID | 允许 NULL FK→t_users | 审核人 |

| created_at | TIMESTAMPTZ | NOT NULL | |
| updated_at | TIMESTAMPTZ | NOT NULL | |

**核心约束**：
- `contract_version_id` 必须引用合同版本表，**申请时锁定版本**
- `ai_suggest_amount` 和 `ai_findings` 是 AI 输出，均为"建议"级别，需人工确认

**审核状态流转**：
```
pending → auditing → audited → paid
    ↑          ↓
  驳回 ← 退回修改
```

## 六、金额与时间处理规范

### 6.1 金额统一用"分"（BIGINT）

| 规则 | 说明 |
|---|---|
| 数据库存储 | `BIGINT`，单位 = 分 |
| 前端展示 | 除以 100，格式化千分位，保留 2 位小数 |
| 用户输入 | 按"元"输入，前端自动 `×100` 后传给后端 |
| API 传输 | `BIGINT`（分），接口文档标注单位 |

**涉及金额的字段清单**：

| 表 | 金额字段 |
|---|---|
| t_projects | total_investment |
| t_data_rows | unit_price, amount |
| t_contract_versions | total_amount |
| t_contract_clauses | payment_amount |
| t_payment_audits | declare_amount, ai_suggest_amount, final_amount, deduct_amount |
| t_secondary_cost_ledger | contract_amount, paid_amount, unpaid_amount |

**前端的金额格式化工具函数**：

```typescript
// 分 → 元，展示
function formatFenToYuan(fen: number): string {
  return (fen / 100).toLocaleString('zh-CN', { minimumFractionDigits: 2 });
}

// 元 → 分，传给后端
function yuanToFen(yuan: number): number {
  return Math.round(yuan * 100);
}
```

### 6.2 时间统一 TIMESTAMPTZ

| 规则 | 说明 |
|---|---|
| 时区 | 全部 `TIMESTAMPTZ` 存储，数据库时区 UTC |
| API 传输 | ISO 8601 格式（如 `2026-07-01T14:30:00+08:00`） |
| 前端展示 | 转换为用户本地时区 |
| 纯日期 | 使用 `DATE` 类型（如 `data_date` `period_start` `valid_from`） |
| 自动维护 | `created_at` 默认 `NOW()`，`updated_at` 由触发器自动更新 |

**updated_at 触发器**：

```sql
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 对每个业务表执行
CREATE TRIGGER trg_{表名}_updated_at
BEFORE UPDATE ON {表名}
FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
```

### 7.1 核心索引清单

| 表 | 索引 | 类型 | 说明 |

| t_contract_clauses | `(contract_version_id, clause_no)` | UNIQUE BTREE | 条款唯一 |
| t_reports | `(project_id, report_type, period_start)` | BTREE | 按项目+类型+周期查报告 |

| t_knowledge_chunks | `(doc_id, chunk_no)` | UNIQUE BTREE | 文档分块唯一 |
| t_knowledge_chunks | `(embedding)` | ivfflat (lists=100) | 向量近似搜索 |
| t_knowledge_chunks | `(tsv)` | GIN | FTS 全文搜索 |
| t_audit_logs | `(table_name, record_id)` | BTREE | 按表+记录查审计 |
| t_audit_logs | `(modified_at DESC)` | BTREE | 按时间查审计（保留 5 年，考虑分区） |
| t_ai_tasks | `(project_id, task_type, status)` | BTREE | 按项目+类型查任务 |
| t_users | `(username)` | UNIQUE BTREE | 登录名唯一 |

### 7.2 索引设计原则

1. **高频查询优先**：针对 `t_data_rows` 建立全覆盖索引
2. **组合索引按最左前缀**：`(project_id, data_date DESC)` 支持 `WHERE project_id=?`
3. **部分索引缩小体积**：`WHERE is_confirmed=true` 的索引只索引已确认数据
4. **向量索引选择**：起步阶段用 `ivfflat`（lists=100），数据量大后切换 `hnsw`
5. **FTS 全文搜索**：`tsv` 字段由触发器自动维护

## 九、SQL 迁移规范

### 9.1 迁移管理

- 使用 Alembic 管理数据库迁移
- 每个迁移文件必须包含 `upgrade()` 和 `downgrade()`
- 迁移命名规范：`{YYYYMMDD_HHMM}_{描述}.py`
- **严禁手动修改已应用的迁移**（只提交新迁移）

### 9.2 迁移模板

```python
"""create t_data_rows table

Revision ID: abc123
Revises: prev_revision
Create Date: 2026-07-02 11:00:00
"""
from alembic import op
import sqlalchemy as sa
from sqlalchemy.dialects.postgresql import UUID, JSONB

revision = 'abc123'
down_revision = 'prev_revision'

def upgrade():
    op.create_table('t_data_rows',
        sa.Column('id', UUID, primary_key=True, server_default=sa.text('gen_random_uuid()')),
        sa.Column('project_id', UUID, sa.ForeignKey('t_projects.id'), nullable=False),
        sa.Column('data_date', sa.Date, nullable=False),
        # ...
        sa.Column('created_at', sa.DateTime(timezone=True), server_default=sa.text('NOW()'), nullable=False),
    )
    op.create_index('idx_data_rows_project_date', 't_data_rows', ['project_id', sa.text('data_date DESC')])

def downgrade():
    op.drop_index('idx_data_rows_project_date')
    op.drop_table('t_data_rows')
```

## 十、扩展与新表规范

### 10.1 新增表检查清单

当后续需要新增业务模块时，按以下清单自检：

- [ ] 表名是否符合 `t_` 前缀 + 英文小写复数命名规范？
- [ ] 是否包含 `id`（UUID PK）？
- [ ] 是否包含 `created_at` 和 `updated_at`（TIMESTAMPTZ）？
- [ ] 需要软删除时，是否包含 `deleted_at`（TIMESTAMPTZ）？
- [ ] 金额字段是否使用 `BIGINT`（分）？
- [ ] 外键是否指向正确的父表？
- [ ] 是否需要唯一约束？
- [ ] 是否在 `t_audit_logs` 中记录了写操作？
- [ ] 字段是否遵循 NULL 约束规则？
- [ ] 是否需要添加索引？
- [ ] 是否需要与 `t_data_rows` 建立数据同步关系？

### 10.2 禁止的操作

| 禁止操作 | 理由 |
|---|---|
| 直接删除业务数据（使用软删除） | 数据不可恢复 |
| DECIMAL 存储金额 | 存在浮点精度问题 |
| 没有 created_at 的表 | 无法追溯创建时间 |
| 没有 updated_at 触发器的表 | 更新时间无法自动维护 |
| 跳过审计日志的写操作 | 操作不可追溯 |
| 直接覆盖旧数据 | 违反版本管理和历史追溯原则 |

## 十一、附录：常用查询模式

### 11.1 按项目 + 时间范围查询数据

```sql
SELECT * FROM t_data_rows
WHERE project_id = :project_id
  AND data_date BETWEEN :start_date AND :end_date
  AND is_confirmed = true
  AND deleted_at IS NULL
ORDER BY data_date DESC, category, item_name;
```

### 11.2 汇总统计（已确认数据）

```sql
SELECT category,
       SUM(amount) as total_amount,
       SUM(actual_quantity) as total_quantity
FROM t_data_rows
WHERE project_id = :project_id
  AND data_date <= :end_date
  AND is_confirmed = true
  AND deleted_at IS NULL
GROUP BY category
ORDER BY category;
```

### 11.3 资料完整性统计

```sql
SELECT doc_category,
       COUNT(*) as total,
       SUM(CASE WHEN parse_status = 'ready' THEN 1 ELSE 0 END) as ready_count
FROM t_knowledge_docs
WHERE project_id = :project_id
  AND deleted_at IS NULL
GROUP BY doc_category;
```

### 11.4 审计历史查询

```sql
SELECT * FROM t_audit_logs
WHERE table_name = 't_data_rows'
  AND record_id = :record_id
ORDER BY modified_at DESC;
```

### 11.5 合同当前生效版本

```sql
SELECT cv.*
FROM t_contract_versions cv
WHERE cv.contract_id = :contract_id
  AND cv.valid_to IS NULL;  -- 部分唯一索引保障只有一个
```

```sql
ORDER BY created_at DESC;
```

### 11.7 AI 任务状态统计

```sql
SELECT status, COUNT(*) as count
FROM t_ai_tasks
WHERE project_id = :project_id
GROUP BY status;
```
