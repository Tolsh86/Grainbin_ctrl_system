## 一、设计原则

本项目采用"**数据驱动、业务对象驱动、AI 辅助、人机协同**"的设计思想。

### 1.1 核心理念

| 原则 | 说明 | 验收判断 |
|---|---|---|
| 一次录入、多模块共享 | 所有业务数据仅录入一次，多个模块共享使用同一份结构化数据 | 同一数据不在两个页面分别录入 |
| AI 旁路增强 | AI 只负责识别、分析、推荐，不直接修改业务数据 | AI 输出默认 `is_confirmed=false` |
| 增量完善 | 所有业务对象允许增量完善，不要求一次性录入完整信息 | 字段为空也能创建业务对象 |
| 文件即原始资料 | 文件永远作为原始资料保存，业务数据来源于解析后的结构化数据 | 文件表与业务表分离 |
| 全程可追溯 | 所有数据均支持来源追溯、历史版本查看以及人工修正 | 每条数据可追溯到来源文件与页面 |

### 1.2 关键设计原则（融入实施计划）

1. **数据落盘优先**：开发优先级 = 数据入库 > 数据展示 > AI 增强。W1 重点在数据可入库可查询，W4 才接入 AI。
2. **旁路增强不替换**：AI 作为辅助层，不替代人工决策。AI 服务不可用时系统不影响手工操作。
3. **数值计算不交 AI**：金额、工程量、比例、累计值全部由 SQL 聚合或 Pandas 计算，LLM 只生成文字描述。
4. **配置驱动**：表单字段、报告模板全部走 JSON Schema 配置，业务方变动仅调配置不动代码。
5. **先治理后展示**：驾驶舱只展示有数据来源、可追溯的指标。汇总统计只取 `is_confirmed=true` 的数据。

## 二、技术架构规范

### 2.1 总体架构

系统采用前后端分离的单体应用架构，自上而下分为五层：

```
[React 18 + Vite 前端] → [Nginx 反向代理] → [FastAPI 业务 + AI 服务]
                                                    |
                                          +---------+---------+
                                          |         |         |
                                      PostgreSQL  MinIO    Redis
                                    (含 pgvector)  (对象存储) (缓存+队列)
```

| 层次 | 技术选型 | 说明 |
|---|---|---|
| 前端应用层 | React 18 + Vite + Ant Design 5 + Formily + ECharts | SPA，动态表单由 JSON Schema 驱动 |
| API 网关层 | FastAPI / Nginx | 统一身份认证、权限路由、审计日志、限流 |
| 业务服务层 | Python FastAPI + SQLAlchemy 2.0（异步）| 单体统一 Python 栈，不引入跨语言 RPC |
| AI 智能体层 | LangGraph + DeepSeek API + PaddleOCR | 异步 Worker，支持降级为纯手工模式 |
| 数据底座层 | PostgreSQL 15+ + MinIO + Redis | pgvector 与业务同库，减少组件依赖 |

### 2.2 接口设计规范

1. **统一前缀**：全部接口以 `/api/v1/` 为前缀。
2. **请求响应**：JSON 格式，Pydantic 模型强校验。
3. **接口分类**：
   - 业务 REST API（CRUD 操作）
   - 知识检索 API（RAG 问答）
   - AI 模型网关 API（兼容 OpenAI 风格，便于模型切换）
4. **异步任务**：AI 耗时操作（PPT 生成、大批量文档解析）走 Celery 异步任务，前端轮询 `t_ai_tasks` 状态。
5. **分页规范**：列表接口统一使用 `page` / `page_size` 参数，响应格式 `{items: [], total: N}`。
6. **错误响应**：统一格式 `{code: "ERROR_CODE", message: "描述", detail: {}}`。

### 2.3 部署规范

- **开发环境**：单机 Docker Compose 一键启动（PostgreSQL + Redis + MinIO + PgAdmin + FastAPI + Vite dev）。
- **生产环境**：单机部署或 2 台应用 + 1 台 DB；不上 K8s（单人维护负担过重）。
- **模型部署**：默认调用 DeepSeek API；数据敏感场景可切换 Qwen2.5 本地部署（Ollama / vLLM）。
- **监控**：Prometheus + Grafana 仅做基础指标（API 响应时间、错误率、队列长度）。

## 三、文件上传规范（★★★★★）

### 3.1 文件永不覆盖

任何上传的文件都不得直接覆盖历史文件。

```
合同.pdf
合同（修改版）.pdf
合同（最终版）.pdf
```

全部保留。数据库始终保存所有历史资料。

**实现要求**：MinIO 对象存储使用 UUID 文件名 + 原始文件名映射，避免同名覆盖。

### 3.2 文件必须保存来源

每一个文件至少保存以下元数据：

| 字段 | 说明 | 示例 |
|---|---|---|
| 所属项目 | 文件归属的项目 ID | project_id |
| 上传人员 | 上传操作的执行人 | user_id |
| 上传时间 | 精确到毫秒 | 2026-07-02T11:00:00+08:00 |
| 文件类型 | 合同/周报/月报/签证/审核/施工资料 | contract |
| 来源单位 | 文件提供单位名称 | XX建设集团有限公司 |
| 文件版本 | 文件版本号（人工或自动） | v1.0 |
| 解析状态 | pending/parsing/ready/failed | pending |
| 文件大小 | 单位：字节 | 1048576 |

### 3.3 文件不等于业务数据

**禁止**：
```
上传合同 → 直接更新合同表
```

**必须**：
```
上传 → 文件表 → AI 解析 → 人工确认 → 业务数据
```

**详细流程**：
1. 用户上传文件 → 写入 `t_knowledge_docs`（或文件关联表），`parse_status=pending`
2. 异步任务解析 → `parse_status=parsing` → 解析完成 → `parse_status=ready`
3. AI 提取结构化数据 → 写入 `t_data_rows`，`is_confirmed=false`
4. 用户在界面上确认/修改 → 确认后 `is_confirmed=true`
5. 已确认数据参与汇总统计、报告生成

## 四、AI 识别规范（★★★★★）

### 4.1 AI 不能直接写数据库

所有 AI 识别结果必须经过人工确认。

**统一流程**：
```
上传 → AI 解析 → 展示识别结果 → 用户修改 → 确认 → 落库
```

**禁止**：
```
AI 识别 → 直接保存
```

**实现约束**：
- AI 识别结果写入 `t_data_rows` 时，`is_confirmed` 字段默认 `false`
- 该数据不参与任何汇总统计、报告生成及驾驶舱展示
- 用户确认操作记录 `t_audit_logs`

### 4.2 AI 必须输出置信度

| 字段 | 识别值 | 置信度 |
|---|---|---|
| 合同金额 | 5000000 | 98% |
| 施工单位 | XX建设 | 81% |
| 负责人 | 张三 | 62% |

**配色规范**：
| 置信度区间 | 颜色 | 含义 |
|---|---|---|
| ≥ 95% | 绿色 | 高可信，建议直接确认 |
| ≥ 80% | 黄色 | 中可信，建议复核后确认 |
| < 80% | 红色 | 低可信，必须人工核实 |

**技术实现**：AI 响应中必须包含 `confidence` 字段，UI 根据置信度渲染颜色标签。

### 4.3 AI 识别结果允许修改

任何识别结果都必须支持：
- **修改**：逐个字段编辑
- **删除**：删除识别出的某条数据
- **新增字段**：在 AI 未识别出的字段上手动补充

不得限制人工修正。

### 4.4 AI 服务降级规范

当 AI 服务（DeepSeek/Qwen API、PaddleOCR）不可用时：

1. **不影响手工操作**：数据上传、手工录入、报告编辑、审核确认等核心功能正常运行
2. **前端感知**：AI 功能按钮显示"AI 服务暂不可用"，点击提示具体原因
3. **降级切换**：系统自动检测 AI 服务健康状态，异常时自动隐藏 AI 辅助功能
4. **恢复检测**：后台定期探测 AI 服务可用性，恢复后自动显示 AI 功能

## 五、数据入库规范（★★★★★）

### 5.1 数据允许为空

业务对象允许部分字段为空。例如：

```
合同
├── 金额           √
├── 施工单位       √
├── 监理单位       NULL（等待后续资料完善）
```

不得因为字段缺失而拒绝创建业务对象。

**实现要求**：数据库表中非核心字段允许 NULL，`NOT NULL` 仅加在必须存在的字段上（如 id、project_id、创建时间等）。

### 5.2 禁止重复创建业务对象

同一个合同，不能因为再次上传资料而重新创建一份合同。

系统应：
- **自动匹配**：根据合同编号、项目 ID 等关键字段自动匹配已有业务对象
- **人工选择**：无法自动匹配时，提示用户选择：

```
⚠️ 检测到可能的重复数据
请选择：
◉ 更新已有合同（合同编号：GC-2026-001）
○ 创建新合同
```

### 5.3 数据更新必须记录历史

```
合同金额：100万 → 120万
```

系统保存：

| 修改前 | 修改后 | 修改人 | 修改时间 |
|---|---|---|---|
| 10000000 | 12000000 | 张三 | 2026-07-02 11:00:00 |

**实现要求**：所有写操作必须写入 `t_audit_logs`，保留完整的 `old_value → new_value` 变更轨迹。不得直接覆盖而不留记录。

## 六、业务对象设计规范（★★★★★）

### 6.1 业务对象体系

所有业务均围绕业务对象展开。业务对象以项目为根节点：

```
项目（t_projects）
├── 合同（t_contracts → t_contract_versions → t_contract_clauses）
├── 数据流水（t_data_rows） ← 系统数据核心底座
├── 报告（t_reports ← t_report_templates）
├── 进度款审核（t_payment_audits）
├── 二类费用（t_secondary_cost_ledger）
├── 知识库文档（t_knowledge_docs → t_knowledge_chunks）
├── 审计日志（t_audit_logs）
└── AI 任务（t_ai_tasks）
```

### 6.2 业务对象设计原则

1. **所有文件均绑定业务对象**，而不是文件之间互相关联。
2. **数据流水表（t_data_rows）是整个系统的事实源**，所有工程资料最终转换为统一结构化数据存储于此表。
3. **四元组原则**：每条数据必须带 `source_type`（来源类型）+ `is_confirmed`（确认状态）+ `source_doc`（来源文件）+ `raw_data`（AI 识别快照）。
4. **汇总统计只取 `is_confirmed=true` 的数据**，未确认数据不进入任何驾驶舱、报告、对比。

### 6.3 技术设计注意

虽然系统采用单表 `t_data_rows` 作为统一数据底座，但为了提升查询性能，高频访问的业务数据（如合同、审核）在业务表中保留"当前态"的快照。业务表与流水表的关系：

```
t_data_rows（全量历史）← 数据源
     ↕ 同步（当前态）
t_contracts / t_payment_audits / ...（当前态，用于高频查询）
```

## 八、数据冲突规范（★★★★★）

### 8.1 冲突检测

当不同资料识别出不同结果时，禁止自动覆盖。

```
合同金额：
├── 合同文件：   1000万
└── 付款申请：   1200万  ← 冲突！
```

### 8.2 冲突处理流程

系统提示：

```
⚠️ 发现数据冲突
字段：合同金额
├── 当前值：1000万（来源：合同.pdf）
└── 新值：   1200万（来源：付款申请表.xlsx）

请选择处理方式：
◉ 保留旧值（1000万）
○ 使用新值（1200万）
○ 手动输入：_________
```

**实现要求**：
1. 冲突数据在 UI 上高亮标记（红色背景 + 冲突图标）
2. 冲突数据 `is_confirmed` 保持 `false`，不参与统计
3. 人工裁决后记录 `t_audit_logs`

## 九、数据来源追溯规范（★★★★）

### 9.1 来源追溯四要素

每一个业务字段均应保存：

| 要素 | 说明 | 存储字段 |
|---|---|---|

| 来源页码 | 数据在文件中的具体位置 | `raw_data` 中的 span 字段 |

```
合同金额：5,000,000 元
└── 来源：合同.pdf → 第3页 → 点击查看原文件
    解析时间：2026-07-01 14:30:00

```

**实现要求**：点击来源文件链接可直接跳转至 MinIO 文件预览页面，并定位到指定页码。

## 十、版本管理规范（★★★★）

### 10.1 业务对象版本管理

所有重要业务对象建议支持版本。

```
合同 V1 → V2 → V3
当前版本：V3
历史版本：只读（查看、不可编辑）
```

### 10.2 合同版本强校验

**DB 层硬约束**：
```sql
-- 同一时间只能有一个生效版本
CREATE UNIQUE INDEX idx_contract_one_active
ON t_contract_versions(contract_id) WHERE valid_to IS NULL;
```

**应用层校验**：
- 进度款申请时强制锁定 `contract_version_id`
- AI 审核 prompt 强制带版本号前缀
- 前端必须显示合同版本号

### 10.3 报告模板版本管理

- 报告模板表 `t_report_templates` 支持版本号（`version` 字段）
- 旧报告永远对应创建时的模板版本
- 新模板只影响新报告
- 模板通过 `is_active` 字段控制启用状态

### 10.4 报告版本管理

- 报告生成后，`data_payload` 保存报告发布时的完整数据快照
- 历史报告不会因为数据库修改而发生变化
- 报告 `report_status` 状态流转：`draft → pending → confirmed → published`

## 十一、数据生命周期规范（★★★★）

### 11.1 资料生命周期

所有资料均具有完整的生命周期状态：

```
上传 → 待解析 → AI识别 → 待确认 → 已确认 → 已引用 → 归档
```

| 状态 | 说明 | 可操作 |
|---|---|---|
| 上传 | 文件已上传至存储 | 等待解析 |
| 待解析 | 文件已入解析队列 | 重新上传、人工录入 |
| AI识别中 | AI 正在解析处理 | 等待 |
| 待确认 | AI 识别完成，等待人工确认 | 确认/修改/删除 |
| 已确认 | 人工确认通过，数据已入正式库 | 追溯、查看 |
| 已引用 | 数据已被报告/审核引用 | 只读 |
| 归档 | 项目完成后归档 | 只读 |

### 11.2 Dashboard 统计

```
数据状态概览
├── 待确认      12份
├── 待解析       3份
├── AI处理中     2份
├── 已确认     128份
├── 已引用      45份
└── 已归档      58份
```

## 十二、报告生成规范（★★★★）

### 12.1 报告不可变

报告不得实时拼接数据库，而应：

```
生成报告
↓
保存 Word / PPT
↓
记录生成时间
↓
记录使用的数据版本（data_payload 快照）
```

保证历史报告不会因为数据库修改而发生变化。

### 12.2 报告生成流程

1. 用户点击"生成报告"
2. 系统检查资料完整性（见第七章）
3. 从 `t_data_rows`（`is_confirmed=true`）汇总数据
4. AI 生成文字描述部分（进度描述、问题分析等）
5. 填充 JSON Schema 定义的表单字段
6. 保存 `data_payload` 快照至 `t_reports`
7. 调用 python-docx / python-pptx 导出文件

### 12.3 模板驱动

- 报告模板由 `t_report_templates.schema_json`（JSON Schema）定义
- 模板支持 5 种字段类型：`text` / `number` / `date` / `formula` / `reference`
- 换模板不换代码，仅修改 JSON Schema

## 十三、Dashboard 设计规范（★★★★）

### 13.1 禁止直接统计业务表

**禁止**：
```
SELECT * FROM t_data_rows → 前端直接聚合
```

**建议**：
```
业务数据 → 统计服务（定时/按需聚合） → 统计结果表（物化视图/缓存） → Dashboard
```

### 13.2 实现方案

- 统计结果写入统计中间表或 Redis 缓存
- 设置缓存过期时间（如 5 分钟）
- Dashboard 页面优先展示缓存数据
- 提供"刷新"按钮触发重新计算

### 13.3 Dashboard 展示内容

| 模块 | 内容 | 数据来源 |
|---|---|---|
| 项目概况 | 在建项目数、本月总产值、完成率 | 统计服务 |
| 工程进度 | 完成率、S 曲线趋势图 | 统计服务 |
| 投资分析 | 合同金额、已完成产值、已支付金额、月度趋势 | 统计服务 |
| 审核统计 | 待审核笔数、审核金额、核减金额、核减率 | 统计服务 |
| 资料完整性 | 资料完整度百分比、各状态数量 | 统计服务 |
| AI 分析摘要 | AI 自动分析摘要 | AI 服务 |

## 十四、前端交互规范（★★★★）

### 14.1 统一交互流程

所有页面统一遵循：

```
上传 → AI 处理中 → 展示结果 → 人工确认 → 保存 → 自动刷新
```

### 14.2 AI 操作状态展示

所有 AI 操作均显示：

| 状态 | 展示方式 | 可操作 |
|---|---|---|
| 处理中 | 加载动画 + 进度提示 + 取消按钮 | 等待 |
| 成功 | 绿色提示 + 数据展示 | 确认/修改 |
| 失败 | 红色提示 + 失败原因 | 重新解析/重新上传/人工录入 |
| 可重试 | 重试按钮 + 原因说明 | 重试 |

不得出现页面无响应。

### 14.3 数据确认交互

提供三档确认交互：

1. **逐条确认**：逐条查看 AI 识别结果，逐个确认或修改
2. **批量确认**：全选后批量确认（仅限置信度 ≥ 95% 的数据）
3. **修改后确认**：编辑字段值后再确认

### 14.4 金额展示规范

- 前端展示时，`BIGINT`（分）自动转换为"元"展示，除以 100
- 金额格式使用千分位分隔符：`5,000,000.00 元`
- 输入时支持直接输入元，前端自动乘以 100 后传给后端

## 十五、异常处理规范（★★★★）

### 15.1 异常类型与提示

| 异常类型 | 提示信息 | 提供操作 |
|---|---|---|
| 文件格式错误 | "不支持的文件格式，请上传 PDF/Word/Excel/图片" | 重新上传 |
| 解析失败 | "文件解析失败，请检查文件是否损坏" | 重新上传、人工录入 |
| OCR 失败 | "图像文字识别失败，请确认图片清晰度" | 重新上传、人工录入 |
| AI 超时 | "AI 服务响应超时，请稍后重试" | 重新解析、人工录入 |
| 数据库保存失败 | "数据保存失败，请稍后重试" | 重试 |
| 字段冲突 | "检测到数据冲突，请确认处理方式" | 人工裁决 |

### 15.2 三种处理方式

所有失败场景必须提供以下三种处理方式之一或多个：

```
◉ 重新解析（重试 AI 操作）
○ 重新上传（更换文件再试）
○ 人工录入（跳过 AI，直接手工填写）
```

### 15.3 全局异常处理

- API 层：统一异常拦截器，返回 `{code, message, detail}`
- 前端层：全局错误提示（message.error / notification）
- AI 层：超时 60 秒自动放弃，标记"解析失败需人工"
- 降级策略：AI 服务不可用时，手工操作完全不受影响

## 十六、权限设计规范（★★★★）

### 16.1 角色权限矩阵

权限应细化到业务对象，避免仅设置"管理员/普通用户"两级权限。

| 角色 | 权限范围 | 典型操作 |
|---|---|---|
| 施工单位 | 上传周报、工程量、施工日志 | 新增、查看自己上传的数据 |
| 建设单位 | 查看全部、确认数据 | 查看所有项目数据、确认 AI 识别结果 |
| 监理单位 | 上传审核意见、查看施工资料 | 审核意见录入、施工资料查阅 |
| 财务 | 维护二类费用、付款记录 | 合同费用管理、付款确认 |
| 管理员 | 系统配置、权限管理 | 用户管理、模板配置、系统设置 |

### 16.2 数据权限

- 基于 `t_users.project_permissions` 字段控制项目级数据可见性
- 跨项目数据隔离，用户只能看到有权限的项目
- 权限判断在 API 层通过中间件实现

### 16.3 审计日志

所有写操作均记录 `t_audit_logs`：

| 字段 | 说明 |
|---|---|
| table_name | 操作的表名 |
| record_id | 操作记录 ID |
| field_name | 修改的字段 |
| old_value | 修改前值 |
| new_value | 修改后值 |
| operation_type | INSERT/UPDATE/DELETE/CONFIRM/EXPORT |
| modified_by | 操作人 |
| source | manual/ai_recognition/system |

审计日志保留 5 年。

## 十七、开发建议（最重要）

整个系统开发过程中应始终遵循以下六条核心原则：

1. **文件只是资料，不是业务数据。** 文件永远作为原始资料保存于 MinIO，业务数据来源于 `t_data_rows` 表中的结构化数据，二者严格分离。

2. **AI 只是助手，最终数据必须由人工确认。** AI 输出默认写 `is_confirmed=false`，不参与任何汇总统计。未经人工确认的数据，不进入驾驶舱、报告和对比分析。

3. **业务对象允许逐步完善，不要求资料一次上传完整。** 所有允许为空的字段不得阻塞业务对象的创建。增量完善是系统的默认工作模式。

4. **任何数据都必须保留来源、版本和修改历史，保证可追溯。** 每条数据携带 `四元组`（source_type + is_confirmed + source_doc + raw_data），每次修改写入 `t_audit_logs`，每个版本通过部分唯一索引强制约束。

5. **所有业务模块共享同一份结构化数据，避免重复录入和数据不一致。** `t_data_rows` 是系统的单一事实源（Single Source of Truth），汇总、对比、报告、驾驶舱全部基于此表。

6. **系统以"项目全过程管理"为核心，而不是简单的文件上传与查询。** 设计目标是通过数据驱动实现从合同签订、施工过程、进度款审核到报告生成的完整业务闭环，AI 在整个过程中起旁路增强作用。
