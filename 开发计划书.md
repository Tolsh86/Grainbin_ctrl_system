## 第一章 会议纪要核心决策提炼

> 来源：《智能纪要：工程过控工作内容及系统优化讨论 2026年6月30日》

| # | 决策 | 对系统设计的直接影响 |
|---|---|---|
| 1 | **节点由用户自己编辑** — 不同公司交付节点不同，系统难以纯数字计算 | 节点模板只是"脚手架"，用户必须能增删改节点、编辑公式、调整触发条件、设置时间偏移 |
| 2 | **系统起推送提醒作用，不做决策** — "不要设计太复杂，不能太依赖，做完之后还是要人为审核" | AI 输出默认 `is_confirmed=false`，支付节点逾期提醒由系统推送但最终由人确认 |
| 3 | **先做定制化，再考虑复用** — "先把自己做好，再看怎么提取模板" | 第一阶段功能全部面向实际业务，不做抽象层；模板/映射等配置化能力从实际使用中反推 |
| 4 | **模板先固定** — 月报/周报产值模板尽快固定 | 报告模板（schema_json）先做 2-3 套实打实的预置模板，后续再开放用户自定义 |
| 5 | **天气影响施工进度** — 接入天气 API 预测进度 | Dashboard 天气卡片 + 项目进度预测（基于历史停工模式 + 天气预报） |
| 6 | **进度看板** — 展示各单位进度，施工单位自行填报 | 项目详情页增加进度看板 Tab，展示各参与单位的完成情况 |
| 7 | **费用类型用户可自定义** — 费用类型（检测/设计/监理/咨询/...）不应写死 | 费用类型下拉支持 tags 模式（选择已有 + 输入新类型），新类型自动落库 |
| 8 | **数据溯源是所有功能的基础** — "所有结果要有原始原因，记录应尽量不被篡改" | 每条数据四元组（source_type + is_confirmed + source_doc + raw_data）不可省略 |

## 第二章 核心设计决策：用户自定义 vs 后端预置

> **这是本次重构最关键的一章。** 每个模块必须明确：哪些东西是后端/系统预置的（开箱即用），哪些东西是用户在使用过程中逐步配置/自定义的（落库存储）。

### 2.1 总表：谁定义什么

| 业务对象 | 系统预置 | 用户自定义 | 存储位置 | 自定义后能否复用 |
|---|---|---|---|---|
| **费用类型** | 检测、设计、监理、咨询、其他（5 个初始值） | 用户输入新费用类型（回车添加） | `t_cost_types`（新增表）或直接从 `t_secondary_cost_ledgers.cost_type` 动态去重 | 全局复用 |
| **节点模板** | 监理模板(5节点)、设计模板(4节点)、检测模板(3节点) | 用户创建自定义模板、修改预置模板的副本 | `t_node_templates`（新增表） | 同一费用类型的合同可复用 |
| **合同支付节点** | 从模板自动生成（选择模板一键生成） | 用户逐个增删改节点、编辑公式、调整日期偏移、修改触发条件 | `t_payment_nodes`（新增表） | 合同间可复制 |
| **节点公式** | 预置公式模板：`=合同金额*比例`、`=固定金额` | 用户自由编辑公式表达式 | `t_payment_nodes.formula_expr` | — |
| **字段映射** | Excel工程量清单、监理月报、进度款申报（3 套预置） | 用户创建新映射、修改已有映射、拖拽调整字段对应关系 | `t_field_mappings`（数据库设计规范中的表） | 映射绑定到项目或全局 |
| **报告模板** | 周报模板、月报模板、审核报告模板（3 套） | 用户修改 schema_json、上传 Word/PPT 模板文件 | `t_report_templates` | 全局 |
| **转换器（Converter）** | passthrough / yuan_to_fen / wan_yuan_to_fen / chinese_date_to_iso / excel_serial_to_date / iso_date / decimal_to_base_unit:ton / category_alias_to_canonical / trim / lowercase（11 个） | 暂不开放自定义转换器（V2 阶段评估） | 后端代码常量 + `t_field_mappings` 引用 | — |
| **项目里程碑** | 无预置 | 用户在项目详情页手动添加关键节点（如：主体封顶、装修完成、竣工验收） | `t_project_milestones`（新增表） | — |

### 2.2 关键原则

```
后端负责：数据结构定义、预置模板、计算引擎（公式解析、金额汇总、日期偏移计算）、校验规则
用户负责：业务内容（节点叫什么、触发条件是什么、付多少钱、什么时候付、字段怎么映射）
系统负责：推送提醒（节点到期前 N 天）、数据汇总、报告自动生成、AI 建议（需人工确认）
```

### 3.2 模块一：项目管理（含进度看板）

**路径：** `/projects`、`/projects/:id`、`/projects/new`

#### 3.2.1 项目列表页

**用户能做什么：**
- 查看自己有权限的项目（卡片网格 + 表格双视图切换）
- 筛选：状态（筹备中/在建/已完工/暂停）、项目经理、地点、关键字
- 新建项目（跳编辑页）
- 每张卡片显示：项目名、编号、合同金额、开工日期、完成率（环形进度条）、状态标签

**系统提供什么：**
- 完成率自动从 `t_data_rows`（`is_confirmed=true`）汇总计算，不由用户手动填写
- 状态标签颜色映射：筹备中=灰、在建=蓝、已完工=绿、暂停=橙
- 卡片排序：默认按创建时间倒序

**接口：** `GET /api/v1/projects`（列表）、`POST /api/v1/projects`（新建）、`PUT /api/v1/projects/:id`（编辑）、`DELETE /api/v1/projects/:id`（软删除）

#### 3.2.2 新建/编辑项目页

**用户填写的信息（全部落库 `t_projects`）：**

| 字段 | 必填 | 说明 |
|---|---|---|
| 项目名称 | ✅ | ≤200 字 |
| 项目编号 | ✅ | ≤50 字，全局唯一 |
| 建设规模 | 否 | 如"6座5000吨级平房仓" |
| 总投资 | ✅ | 用户输入"元"，前端 `×100` 转"分"存库 |
| 建设单位 | 否 | 允许后续补充 |
| 监理单位 | 否 | 允许后续补充 |
| 设计单位 | 否 | 允许后续补充 |
| 施工单位 | 否 | 允许后续补充 |
| 项目经理 | 否 | 下拉选 project_manager 角色用户 |
| 项目地点 | 否 | 用于天气卡片定位 |
| 计划开工日期 | ✅ | DatePicker |
| 计划完工日期 | ✅ | DatePicker |
| 项目状态 | 系统默认 | 新建默认"筹备中" |
| 备注 | 否 | TEXT |

**设计决策：** 参建单位（建设/监理/设计/施工）允许为空 — 符合会议"增量完善"原则，不要求资料一次上传完整。

#### 3.2.3 项目详情页（含进度看板）

**页头：** 项目名、编号、状态标签、完成率、快捷操作按钮

**Tab 1：概览（默认）**
- 基本信息卡片（左侧 2/3）
- 投资概览卡片（合同总额 / 累计完成 / 累计已付 / 待付，4 个统计数字，从 t_data_rows 汇总）
- 天气卡片（调用 Open-Meteo，显示项目所在地 7 天天气预报）
- **项目里程碑时间轴**（右侧 1/3 或下方通栏）

**Tab 2：进度看板（新增 ★）**

**进度看板的核心设计：**

| 要素 | 谁填 | 存储位置 | 说明 |
|---|---|---|---|
| 各单位进度卡片 | 各角色用户自行填报 | `t_progress_reports`（新增表） | 施工单位填施工进度、监理单位填监理工作进度等 |
| 整体进度百分比 | 系统根据 t_data_rows 汇总 | 实时计算 | 不由用户手动填 |
| 项目里程碑 | 项目经理在项目详情页手动添加 | `t_project_milestones`（新增表） | 每个节点：名称、计划日期、实际完成日期、状态 |
| 里程碑时间轴 | 系统根据里程碑数据渲染 | 前端组件 | 水平时间轴 + 节点状态标记 |

**`t_project_milestones`（新增表）设计：**

```sql
CREATE TABLE t_project_milestones (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID NOT NULL REFERENCES t_projects(id),
    seq INTEGER NOT NULL,                         -- 排序号
    name VARCHAR(200) NOT NULL,                   -- 里程碑名称："主体封顶"
    planned_date DATE,                            -- 计划完成日期
    actual_date DATE,                             -- 实际完成日期
    status VARCHAR(20) DEFAULT 'pending',         -- pending/completed/delayed
    description TEXT,                              -- 备注
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**`t_progress_reports`（新增表）设计：**

```sql
CREATE TABLE t_progress_reports (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID NOT NULL REFERENCES t_projects(id),
    reporter_id UUID NOT NULL REFERENCES t_users(id),
    unit_type VARCHAR(30) NOT NULL,               -- constructor/supervisor/designer/owner
    content JSONB NOT NULL DEFAULT '{}',          -- 各分项进度描述 + 百分比
    report_date DATE NOT NULL,                    -- 填报日期
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**进度看板的 Tab 内 UI 构成：**
1. **顶部** — 日期范围选择 + "本周"快捷按钮
2. **中部** — 4 列卡片（施工/监理/设计/建设），每列展示该单位当前填报的最新进度，底部有"填报进度"按钮（仅对应角色可见）
3. **底部** — 项目整体里程碑时间轴（水平滚动，当前节点高亮，已完成节点打勾）

**Tab 3-6：** 数据 / 报告 / 审核 / 文档 — 跳转到对应模块并自动过滤本项目

### 3.3 模块二：数据中心（系统核心）

**路径：** `/data/*`

#### 3.3.1 数据上传页 `/data/upload`

**上传前的用户操作（必填）：**

```
1. 选择项目          → projectStore 同步
2. 选择业务周期       → 类型（周报/月报/进度款/审核资料/其他）+ 起止日期
3. 选择字段映射       → 下拉显示可用映射，可跳管理页新建
4. 拖拽/点击上传文件  → 文件列表预览（文件名/大小/状态）
5. 预览第一个文件表头 → 确认字段对应无误
6. 点击"开始上传"    → 后端创建 t_ingest_batches，异步解析
```

**用户自定义点：字段映射**

字段映射是用户在上传前选择（或在此之前创建）的配置。核心交互：

- 映射编辑器（`/data/mapping/:id/edit`）是一个**表格**：

```
┌──────────────────────────────────────────────────────┐
│  映射名：Excel 工程量清单                               │
│  适用项目：1-6号粮仓新建工程（或"通用"）                 │
│  文件格式：xlsx/xls                                    │
│  表头行号：1                                           │
│  Sheet 索引：0                                         │
├──────────┬──────────────┬────────────────┬───────────┤
│  用户表头  │  系统字段      │  转换器         │  操作      │
│  (输入)   │  (下拉选择)    │  (下拉选择)      │           │
├──────────┼──────────────┼────────────────┼───────────┤
│ 施工日期  │  data_date    │ iso_date       │  🗑  ↕    │
│ 分项名称  │  item_name    │ passthrough    │  🗑  ↕    │
│ 完成数量  │ actual_qty    │ passthrough    │  🗑  ↕    │
│ 单价(元) │  unit_price   │ yuan_to_fen    │  🗑  ↕    │
│ 合价(元) │  amount       │ yuan_to_fen    │  🗑  ↕    │
│ 单位     │  unit         │ passthrough    │  🗑  ↕    │
│ 费用类别  │  cost_type    │ category_alias │  🗑  ↕    │
│          │               │ _to_canonical  │           │
├──────────┴──────────────┴────────────────┴───────────┤
│  [+ 新增一行映射]                                       │
│                                                        │
│  [保存]  [保存并测试（上传示例文件预览解析结果）]  [取消]   │
└──────────────────────────────────────────────────────┘
```

**用户自定义 vs 系统预置：**
| 项目 | 谁定义 |
|---|---|
| 系统字段列表（data_date, category, item_name, ... 共 12 个） | 后端预置（不可变） |
| 转换器列表（11 个预置转换器） | 后端预置 |
| 映射名、适用项目、表头行号、Sheet 索引 | 用户自定义 |
| 用户表头 → 系统字段的对应关系 | 用户自定义（从下拉选择） |
| 每个字段使用的转换器 | 用户自定义（从下拉选择） |
| 3 套预置映射（Excel工程量清单/监理月报/进度款申报） | 系统预置，用户可复制修改 |

**上传后的流程：**
```
文件上传 → MinIO 存储 → 后端创建 t_ingest_batches(status=pending)
→ Celery 异步解析(pending→parsing→normalized→validated)
→ 前端轮询批次状态 → 状态变为 validated 后跳批次详情
```

#### 3.3.2 批次列表页 `/data/batches`

**功能重点：**
- 表格列：批次号、文件名、项目、状态（彩色 StatusTag）、总行数/成功/错误（红徽标）、质量分（进度条）、周期、上传者、时间、操作
- 状态枚举：pending → parsing → normalized → validated → committing → committed / rolled_back / failed
- 单条操作：查看详情 / 撤回（仅 committed）
- 批量操作：批量撤回（多选 + 二次确认）
- 筛选：项目、状态（多选）、文件格式、周期、上传者、关键字

**行内编辑的关键行为：**
- 点击单元格 → 单元格变为输入框 → 修改值 → 回车保存 → 行自动重校验 → 校验标记更新

- 校验通过的编辑行标记为 `validation_flags.ok`

**整批确认入库的行为：**
- 仅校验通过的行可以被确认
- `t_ingest_rows` → `t_data_rows`（数据迁移）
- `t_data_rows.is_confirmed = true`
- 批次状态 → `committed`
- 记录 `t_audit_logs`（operation_type=CONFIRM）
- **此操作不可逆（除非整批撤回）**

### 3.6 模块五：二类费用（★ 本次重构重点）

**路径：** `/secondary/*`

> **会议纪要核心诉求：** 不同公司的交付节点不同，系统难以通过纯数字计算，需让用户手动编辑节点并输入百分比/公式，系统按时间线推送提醒，但不具备决策功能。

#### 3.6.1 公司合作关系的建模

**"各公司的合作"是哪种角色类型？**

在二类费用场景中，与项目合作的各公司（监理公司、设计院、检测中心、咨询公司等）**不是系统角色**，而是**合同乙方**。它们之间的关系是：

```
项目（1个）
  ├── 二类费用合同1：监理 → 建科监理公司 → 费用类型=监理
  ├── 二类费用合同2：设计 → 粮食设计院   → 费用类型=设计
  ├── 二类费用合同3：检测 → 省检测中心   → 费用类型=检测
  ├── 二类费用合同4：咨询 → 造价咨询公司 → 费用类型=咨询
  └── 二类费用合同5：其他 → XX公司       → 费用类型=用户自定义
```

**数据模型对应：**
- `t_secondary_cost_ledgers.company_name` → 承接单位名称（乙方）
- `t_secondary_cost_ledgers.cost_type` → 费用类型（监理/设计/检测/咨询/用户自定义）
- `t_secondary_cost_ledgers.payment_nodes` → JSONB 支付节点列表

**费用类型是可扩展的：** 系统预置 5 个（监理/设计/检测/咨询/其他），用户通过 Select 的 tags 模式输入新类型，新类型自动落库并可全局复用。

> **注意：** 数据库设计规范 V1.0 中 `t_secondary_cost_ledgers` 的 `payment_nodes` 是 JSONB，但实际开发中建议拆分为独立表 `t_payment_nodes`，以便于：
> - 单个节点的独立查询和更新
> - 节点级别的审计日志
> - 节点到期提醒的精确查询
>
> JSONB 方案（当前数据库设计规范）作为第一阶段的快速实现方案；独立表方案作为第二阶段重构目标。

**公式编辑语法（用户可自由输入）：**

| 公式写法 | 含义 | 计算结果示例 |
|---|---|---|
| `=合同金额*0.30` | 合同总金额 × 30% | 合同 ¥2,800,000 → ¥840,000 |
| `=合同金额*0.15` | 合同总金额 × 15% | 合同 ¥2,800,000 → ¥420,000 |
| `=1200000` | 固定金额（单位：元） | ¥1,200,000 |
| `=合同金额*0.30+50000` | 合同金额 × 30% + 5万（V2 阶段支持） | — |

**第一阶段支持的变量：** `合同金额`（当前合同的 contract_amount）

**时间偏移规则：**
- `offset_days=0` → 计划日期 = 合同起始日期
- `offset_days=30` → 计划日期 = 合同起始日期 + 30 天
- `offset_days=730` → 计划日期 = 合同起始日期 + 730 天（如质保期满）

- 删除（二次确认）
- 从模板导入（选择一个节点模板，覆盖当前合同的节点列表）
- 从其他合同复制（选择一个已有配好节点的合同，复制其全部节点）

**用户自定义 vs 系统预置：**
| 项目 | 谁定义 |
|---|---|
| 节点模板（3 套预置） | 系统预置，用户可修改/复制/新建 |
| 节点名称 | 用户自定义 |
| 节点公式 | **用户自定义**（系统提供公式解析器） |
| 公式计算 | **后端计算**（解析公式 → 代入变量 → 返回金额） |
| 触发条件 | 用户自定义（文本描述） |
| 时间偏移天数 | 用户自定义 |
| 计划支付日期 | **系统自动计算**（合同起始日 + 偏移天数） |
| 实际支付日期 | 用户填写（支付时） |
| 节点状态 | 系统自动（pending=未到期 / overdue=逾期 / paid=已支付） |

**提醒逻辑（后端定时任务）：**
1. 每天 9:00 扫描 `t_payment_nodes`
2. `planned_date - NOW() <= reminder_days` 且 `status != 'paid'` → 生成提醒
3. 站内信写入 `t_notifications`，邮件/短信通过 Celery 异步发送

CREATE TABLE t_payment_nodes (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    contract_id UUID NOT NULL REFERENCES t_secondary_cost_ledgers(id) ON DELETE CASCADE,
    seq INTEGER NOT NULL,                         -- 排序号
    name VARCHAR(200) NOT NULL,                   -- 节点名称

    formula_expr VARCHAR(500) NOT NULL,           -- 公式表达式："=合同金额*0.30"

    offset_days INTEGER DEFAULT 0,                -- 从合同起始日偏移天数
    planned_date DATE,                            -- 计划支付日期
    actual_date DATE,                             -- 实际支付日期
    status VARCHAR(20) DEFAULT 'pending',         -- pending/overdue/paid
    paid_amount BIGINT DEFAULT 0,                 -- 实付金额（分）
    payment_evidence VARCHAR(500),                -- 支付凭证 MinIO 路径
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(contract_id, seq)
);

## 附录 B：与《数据库设计规范 V1.0》的差异说明

| 差异点 | V1.0 | V2.0（本文档） | 原因 |
|---|---|---|---|
| 支付节点存储 | `t_secondary_cost_ledgers.payment_nodes` JSONB | 建议拆出 `t_payment_nodes` 独立表 | 支持节点级独立查询、索引、审计、提醒 |

| 进度填报 | 无 | 新增 `t_progress_reports` | 会议纪要要求各单位自行填报进度 |
| 费用类型 | 无字典表，直接用字符串 | 可选新增 `t_cost_types` | 用户自定义费用类型需要管理 |
